3. Beginner Training Sprint (30 minutes)

This is the "zero to hero" portion. The goal is not mastery, but enough understanding to start building.

3A. Live Demo: Build a Skill in 5 Minutes (5 min)

Build a skill live on screen. We're building a Weekly Status Update skill — something everyone does, whether you're a nurse, wealth advisor, security analyst, or admissions counselor.

Step 1: Create the folder

mkdir -p ~/.openclaw/workspace/skills/weekly-status-update

Step 2: Write SKILL.md

---
name: weekly-status-update
description: Generate a structured weekly status update from rough notes about what you accomplished, what's next, and any blockers. Use every Friday.
---

Take the rough weekly update notes provided and turn them into a structured status update.

## Required Sections
1. **Accomplishments This Week** (bullet points, max 5)
2. **Priorities for Next Week** (bullet points, max 5)
3. **Blockers & Risks** (anything that needs attention or escalation)
4. **Key Metrics / Numbers** (any quantifiable results)

## Rules
- Don't invent accomplishments not in the raw notes
- Keep the total output under 200 words
- Flag anything time-sensitive with ⚠️

Step 3: Test it

/new
"Here's what I did this week: [paste rough notes]"

Done. That's the entire cycle. Emphasize: from zero to working skill in under 5 minutes.

3B. How We Just Built That (8 min)

We just built a skill live. Now let's reverse-engineer what we actually did. The Record-Transcribe-Refine Method:

  1. Record how you do something right now (screen recording, voice memo, or write down the steps)
  2. Transcribe what you recorded into plain language
  3. Feed it to AI and say: "Turn this into a SKILL.md for [platform]"
  4. Review and refine the output — fix gaps, add edge cases, test it
  5. Ship it and iterate based on how it performs

Why this works: You already have workflows you do manually. The skill is just capturing that expertise in a format an AI can follow. You're not inventing from scratch — you're documenting what you already know.

Note: Record-Transcribe-Refine is the recommended beginner onboarding path. As you build more skills, you'll likely start writing SKILL.md directly and iterating based on how the agent performs.

3C. Anatomy of a Skill (10 min)

The Chef & Recipe Model:

  • A general-purpose AI agent is the Chef — it knows how to cook
  • An AI Skill is the Recipe — a modular, reusable package of instructions, scripts, and domain expertise that tells the chef exactly how to execute a specific dish
  • Without skills: the chef guesses. With skills: the chef executes precisely.

Key insight: Skills are primarily structured instructions written in markdown, optionally with supporting executable scripts, reference documents, and templates. Think of them as "onboarding documents for an AI."

Important caveat: Unlike recipes, skills are non-deterministic. The same skill may produce slightly different output each time because it depends on an AI model's interpretation. This is normal and expected — treat skills as rough drafts that get better with iteration, not as guaranteed outputs.

A skill is a folder with a required SKILL.md file and optional supporting resources:

my-skill/
├── SKILL.md              (required - the recipe)
├── scripts/              (optional - executable code)
│   └── validate.sh
├── references/           (optional - docs loaded on demand)
│   └── api-docs.md
└── assets/               (optional - templates, images)
    └── template.md

SKILL.md has two parts:

1. YAML Frontmatter (metadata):

---
name: my-skill
description: What this skill does and when to trigger it. This is how the agent discovers the skill.
---

2. Markdown Body (instructions):

# My Skill

Steps to execute when this skill is active:

1. Do X
2. Then do Y
3. Return result in format Z

That's it. A working skill can be as little as a 10-line markdown file.

What is a terminal? A text-based way to tell your computer to do things. You'll use it to create folders and files for your skill.

  • macOS: Press Cmd+Space, type "Terminal", press Enter
  • Windows: Press the Windows key, type "PowerShell", press Enter

3E. Common Gotchas (3 min)

Top mistakes beginners make:

  • Skills should DO things, not TEACH things. Don't dump domain knowledge into instructions — give focused execution steps.
  • Keep scope narrow. One skill, one job. Don't try to solve your whole workflow in one skill.
  • Test early with real inputs. Don't wait until it's "perfect." Feed it real data and see what happens.
  • The description field matters. That's how the agent discovers your skill. If it's vague, the skill won't trigger.
  • First output won't be perfect. Iterate on the instructions, not the prompt.

What Skills Can't Do: Skills can't guarantee consistent output (non-deterministic), can't execute actions on their own without an agent runtime, and can't replace human judgment. They are force-multipliers, not autonomous workers.

7. Advanced Topics (For Interested Participants)

These are not covered in the training sprint but are mentioned here as next steps for participants who want to go deeper after the hackathon.

7A. Skill Chaining

Skills can reference other skills in their instructions. When one skill mentions "use the X skill," the agent will load and follow that skill as a sub-step.

Example: A weekly-report skill that chains:

  1. Use the meeting-notes skill to format recent meeting notes
  2. Use the github-issues skill to summarize open issues
  3. Synthesize both into a weekly status report

7B. Multi-Agent Orchestration (OpenClaw)

OpenClaw supports spawning sub-agents — isolated background agent runs that complete tasks and report back. Key concepts: Sub-agents, Task Flow, and Lobster (deterministic pipeline runtime with built-in approval gates).

7C. Progressive Disclosure with References

The references/ directory lets you attach supplementary docs that the agent loads only when needed.

brand-voice-checker/
├── SKILL.md              (instructions + "load references/brand-guide.md for rules")
├── references/
│   ├── brand-guide.md    (tone rules, banned words)
│   ├── examples.md       (good/bad examples)
│   └── style-tokens.md   (phrases to use/avoid)

7D. Publishing to ClawHub
